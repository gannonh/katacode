# 2026-06-20 upstream sync closure

## Goal

Land the first episodic upstream sync for Kata Code: absorb 80 commits from `pingdotgg/t3code` (baseline `708d5383` → upstream tip `97e5cd3bf`) as a single bulk merge on the integration branch `upstream-sync-2026-06-20`, then complete the sync-scoped follow-up work the merge causes, then land on `main`. The merge and its closure work are one accepted unit: the branch does not merge to `main` until both pass.

**Policy source of truth:** [ADR 0003 — episodic upstream sync](/adrs/0003-episodic-upstream-sync.md). **Runbook:** [upstream-sync skill](../../.agents/skills/upstream-sync/SKILL.md) and [upstream-sync guide](/guides/upstream-sync.md). **Fork baseline & divergence log:** [FORK.md](../../FORK.md).

## Current state

- Merge-base of `main` and `upstream/main` is `708d5383a` (matches the FORK.md baseline). Fork has diverged 82 commits; upstream has diverged 80 commits since baseline. True divergence; no fast-forward possible.
- Draft classification (`sync-plan.md`, regenerated by `.agents/skills/upstream-sync/scripts/classify-upstream.ts`): **63 take / 2 reject / 11 defer / 4 review**.
- Predicted conflict surface (`conflict-zones.ts`): 649 intersecting paths, 450 in high-blast zones — 196 `apps/server`, 178 `apps/web`, 66 `apps/desktop`, 5 `packages/shared`, 2 `package.json`, 2 `packages/contracts`, 1 `pnpm-lock.yaml`.
- Pre-merge divergence log already recorded in FORK.md at commit `cbe4b46e3`: 2 rejects (marketing homepage `3bdaa6e1`, EAS workflow non-ported portion of `9544e72d`) and 1 ported improvement (EAS preview label gate from `9544e72d`, hand-ported into `.github/disabled/mobile-eas-preview.yml`, Blacksmith runner not ported).
- Branch state: `upstream-sync-2026-06-20`, clean working tree, 3 commits ahead of `main` (divergence log + two skill restructure commits). Not yet merged.

## Constraints

- Bulk merge only; no cherry-pick split (the 39 `[codex]` Effect service refactor commits depend on each other and break compilation if taken individually).
- All fork-policy resolution rules in the runbook's Step 3 apply: restore `Kata Code` / `KATACODE_*` / `katacode://` / `@kata-sh/code-*` / `com.katacode.app` branding on product surfaces; never reintroduce `@t3tools/*` or `T3CODE_*` without an explicit FORK.md decision; do not hand-merge `pnpm-lock.yaml` (regenerate via `vp i`); keep fork rebrand test fixtures upstream-shaped where required (`octocat/t3code`).
- Keep fork wire identifiers intact: `kata_relay`, `kata-mobile`, `kata-web`, `/.well-known/kata/environment`, `/api/kata-connect` (see FORK.md deferred wire-identity section).
- Verify gates required by AGENTS.md: `vp check` and `vp run typecheck` must pass before landing. `vp run test`, `vp run release:smoke` (if release paths touched), `vp run lint:mobile` (if native mobile touched) as applicable.
- Closure work lands on the integration branch. `main` is not touched until closure is complete.

## Out of scope

- Cross-sync deferred work (e.g. Mobile EAS production release, Connect open signups, marketing Connect pages) — these belong in `docs/specs/deferred-work.md`, not this spec.
- Changing fork sync policy (ADR 0003) or the runbook itself.
- Contributing changes back upstream.

Audit whether any merge-exposed case should have produced a deterministic Take/Reject/Defer rather than `review`, and add a rule to `scripts/rules.ts` if so. ## Risks and mitigations

- **Desktop auth migration is large and modify/delete.** The fork has branding commits on the files upstream deletes. Mitigation: dedicated resolution pass; accept upstream's deletions; re-apply branding on the new files; regenerate tests.
- **Clerk build-injection may require release-workflow changes beyond a define rename.** Mitigation: Task 2 is a hard gate; Build stops and asks if the define cannot be sourced without a long-lived secret.
- **`infra/relay/` heavy intersection (20+ files both sides).** Mitigation: resolve by zone, keeping kata wire identifiers; resolve in the same pass as the `[codex]` relay commits.
- **A branded desktop build may fail to ship without the publishable key.** Mitigation: Task 2 acceptance criterion requires a successful branded build with the define resolved before landing.
- **Merge may surface more branding regressions than expected (450 high-blast files).** Mitigation: Task 1 scan is a hard gate; do not land until the scan is clean outside known exemptions.
